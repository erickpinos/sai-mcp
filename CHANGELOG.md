and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added

- **`sai_send` (withdraw / empty wallet)**: a new on-chain write tool that transfers native NIBI or **any ERC20** out of the signer wallet to another address. `token` accepts `"nibi"`, `"usdc"` (shorthand for the collateral contract), or any ERC20 contract address (`0x...`, e.g. stNIBI or another deposited token) — `decimals` and `symbol` are read live from the contract, so it isn't limited to USDC. The recipient may be an EVM `0x...` or a Nibiru `nibi1...` bech32 address (decoded to EVM via the new `bech32ToEvm` / `normalizeRecipientToEvm` helpers). Pass `amount` for a specific quantity or `all: true` to sweep the full balance (the NIBI sweep keeps back only its own gas). Like the trade tools it defaults to a dry-run (`confirm: false`) that resolves the recipient, reads balances, and estimates gas without broadcasting. Plain transfers are **not** gas-sponsored (sponsorship is specific to the Sai perp contract): although `eth_gasPrice` reports `0`, the chain enforces a minimum gas price (~1e12 wei/gas, ≈0.05 NIBI per transfer) deducted from the sender, so the wallet must hold native NIBI to send — even to withdraw USDC. The tool floors the effective gas price to that minimum (`MIN_GAS_PRICE_WEI`) so the broadcast is accepted and the dry-run's `funded` flag is honest (`false` when the wallet has no NIBI for gas).
- **Offboarding + key-export guidance in server `instructions`**: the model is now told how to answer "how do I get my funds out?" (close positions, then `sai_send` each token, sweeping NIBI last) and "how do I get my private key out?" — there is intentionally no tool that returns the secret and the assistant must never print it; the user retrieves it themselves from the `0600` keystore file (`~/.sai-mcp/wallet.json`, override `SAI_KEYSTORE`) or their own `SAI_MNEMONIC` / `SAI_PRIVATE_KEY` config.
- **`keygen --save` + local keystore**: `sai-mcp keygen --save` persists the generated mnemonic to `~/.sai-mcp/wallet.json` (mode `0600`, path overridable via `SAI_KEYSTORE`) and prints **only the public addresses** — the mnemonic and private key are deliberately never written to stdout in this mode, so an AI agent can run setup on the user's behalf without the seed reaching the model/transcript. The keystore is the backup; the server reads it live on each tool call whenever neither `SAI_MNEMONIC` nor `SAI_PRIVATE_KEY` is set (no restart, no config edit). Env vars still take precedence. Loose file permissions trigger a `chmod 600` warning. The plain `keygen` (no flag) still prints the secret for manual paste into a config `env` block and is meant to be run by a human in a terminal, not by an assistant.- **Onboarding/funding guidance in server `instructions`**: the model is now told never to request or print a seed phrase in chat (direct the user to `keygen --save`), and that a freshly created wallet is empty — it must surface the funding address, wait for the user to send NIBI + USDC, and only place a trade once `sai_get_wallet_info` shows non-zero balances, rather than attempting a trade against an unfunded wallet.
- **Tool annotations** on all 25 tools: read tools get `title`, `readOnlyHint`, `openWorldHint`; the 5 write tools additionally get `destructiveHint` and `idempotentHint: false`. Lets clients distinguish the 20 read tools from the on-chain write tools.
- **Structured output**: tools now return `structuredContent` alongside the human-readable text. The write tools and `sai_get_wallet_info` advertise a typed `outputSchema`; read tools return their payload under a `result` envelope.
- **Server `instructions`**: the initialize response now carries the protocol conventions (micro-units, market IDs, funding-rate formula) and the write-tool safety model, so a connected model has the context without being told.
- **MCP Resources**: `sai://guide` (conventions + safety model), `sai://markets` and `sai://tokens` (live mainnet snapshots), and `sai://schema` (GraphQL query roots and fields, derived from live introspection of the keeper endpoint, with a curated fallback).
- **MCP Prompts**: `analyze_trader`, `market_overview`, `vault_yield_report`, and `protocol_health` analysis templates that drive the relevant tools.
- **Optional Streamable HTTP transport**: set `SAI_HTTP_PORT` (or pass `--http [port]`) to serve over HTTP instead of stdio. Stateful sessions (capped), binds to `127.0.0.1` with DNS-rebinding protection, a request body size limit, and an optional `SAI_HTTP_TOKEN` bearer gate (constant-time check) enforced whenever a signer is configured. `SAI_HTTP_HOST` overrides the bind address and `SAI_HTTP_ALLOWED_HOSTS` widens the host allowlist for non-local binds. Added the `start:http` npm script.
- **EVM address support** on all address filters: `sai_get_trader_trades`, `sai_get_trader_history`, `sai_get_user_portfolio`, `sai_get_fee_tier_progress`, `sai_get_referrals`, `sai_get_referral_for_trader`, and the LP deposit/withdraw depositor filters now accept an EVM hex address (`0x...`) in addition to a Nibiru bech32 address (`nibi1...`). The keeper only indexes by bech32, so a raw `0x` previously returned an empty result; the server now converts `0x` to bech32 (via `normalizeTraderAddress`) before querying, and rejects malformed hex with a clear error.
- **EVM share-token support** on vault filters: `sai_get_vault_stats`, `sai_get_deposit_history`, and `sai_get_withdraw_requests` now accept a vault's EVM share-token address (`sharesERC20`, `0x...`) in addition to its bech32 CosmWasm address. A vault's two forms are unrelated byte-wise, so the `0x` is resolved against the live vault list (cached per network, refetched on a miss); passing a collateral-token `0x` returns a targeted error.

### Changed

- Server construction is now a `createServer()` factory shared by both the stdio and HTTP transports, so tools, resources, and prompts are registered in exactly one place.
- Wallet onboarding is keystore-first and split by intent. New wallet: `sai-mcp keygen --save` (the recommended, in-session-safe path). Existing wallet: there is intentionally **no import command** (a CLI import would route the seed through shell history or, via an AI session's `!` prefix, into the transcript); instead the user writes the `0600` keystore JSON themselves in their own terminal/editor (`~/.sai-mcp/wallet.json` with `{"mnemonic": "..."}` or `{"privateKey": "0x..."}`, override `SAI_KEYSTORE`), and the server picks it up live. The README, the "No signer configured" error, and the server `instructions` all reflect this; `SAI_MNEMONIC` / `SAI_PRIVATE_KEY` are demoted to a hosted/Docker/CI note, and write-tool descriptions no longer foreground them ("Requires a configured signer").
- Server `instructions` now tell the model to ASK the user before creating a wallet, never to accept or print a seed phrase / private key in chat, and never to write the keystore on the user's behalf from a shared secret. The onboarding funding step drops the stale "send NIBI for gas" line (trading is gasless): fund USDC only.
- README: scoped the "refuses to broadcast if gas estimation fails" note to the management tools, since `sai_open_trade` deliberately falls back to a fixed gas limit and broadcasts anyway (flagged via `broadcastWithFallbackGas`).

## [0.1.0]

### Added

- Initial release. MCP server exposing the `sai-keeper` GraphQL indexer, REST stats API, and candles API as tools: markets, trader positions and history, portfolios, fee tiers, LP vaults, oracle prices, referrals, leaderboards, protocol stats, yield opportunities, OHLCV candles, and a `sai_graphql_query` escape hatch.
- Opt-in on-chain write tools (`sai_open_trade`, `sai_close_trade`, `sai_update_tpsl`, `sai_update_leverage`, `sai_get_wallet_info`) gated on a signer (`SAI_MNEMONIC` / `SAI_PRIVATE_KEY`), defaulting to dry-run, with operator-set trade guard rails.
